  2. 都变成从"包"的相对引用，而不是绝对路径引用。此外相对路径的顶层（chat，parsing）问题比较大，应变成lazymind，即from lazymind.algorithm.chat import xxx

  2. 开发"新算法的上线和切流"的机制。具体的：
    - 一个知识库可以配置多个算法，这些算法都会被离线解析所用到。普通用户对算法不可见，管理员在开发者模式下，可以看到算法。
    - 在chat的时候，有一个"router模块"用于调控chat究竟选择哪个算法，或者是双回复。默认情况下，会选择主算法。

    - 灰度测试即配置router，对指定的知识库进行切流；灰度通过后，将新算法设置为主算法。30天内支持回滚，过期删除旧的算法。

### A — UI/UX 体验优化

#### A1. Chat 界面 Mermaid 渲染

**背景**：当前 chat 回复中的 mermaid 代码块以纯文本展示，无法直观呈现流程图、时序图等。

### C — 用户与权限管理

#### C1. system-admin 保护规则

**需求细化**：
- `system-admin` 角色为系统内置角色，不可删除、不可重命名。
- 拥有 `system-admin` 角色的用户：

- 前端在用户管理界面对 system-admin 用户的操作按钮（禁用、修改角色、删除）置灰并 tooltip 说明原因。

#### C2. 权限模型（参考 3.5 设计）

**需求细化**：
该模块的目标是构建一套清晰、统一且可扩展的权限管理体系，覆盖“用户-用户组-知识库”三层关系，实现灵活的协作与访问控制。整体设计参考类似GitLab的模式，但在复杂度上做了适当收敛，确保系统易于理解与落地。权限模型的核心思想是：用户可以通过“直接授权”或“组间接授权”两种方式获得资源权限，系统通过统一规则计算最终权限，从而兼顾灵活性与可控性。
用户在系统中的核心使用流程为：创建用户组 → 邀请成员并设置组内权限 → 在知识库中授权用户组或个人 → 用户基于最终权限进行操作。该流程强调“以组为中心”的协作方式，即推荐优先通过用户组进行权限分发，而非逐个用户配置。关键交互要点包括：
1. 任意用户都可以创建用户组，一个用户可以创建或加入多个组，组分为“公开”和“私有”，公开组可被搜索和发现，私有组仅成员可见
2. 组支持直接邀请用户加入，用户无需确认即成为成员，邀请时需指定权限（管理/写/读），并在成员列表中支持后续修改
3. 用户可以在“我的组”页面查看自己所属的所有组及对应权限，并可快速进入组进行管理
4. 知识库支持授权给“用户”或“用户组”，授权时需指定权限级别（管理/写/读），并支持随时调整或撤销
5. 在知识库权限页面，用户可以查看“权限来源”，系统需清晰展示某个用户的权限是来自哪个组或个人授权
权限模型采用三种基础权限：管理（admin）、写（write）、读（read），并统一适用于“组内权限”和“知识库权限”。其中管理权限包含成员管理与授权能力，写权限包含资源操作能力（如编辑知识库、触发解析等），读权限仅允许查看与使用。权限设计遵循“最小充分原则”，即仅保留三种核心权限，避免复杂组合带来的理解成本。
在权限计算上，系统定义明确规则：用户在某个知识库的最终权限 =
max(min(组1在知识库的权限，用户在组1的权限)，min(组2在知识库的权限，用户在组2的权限)，...，用户在知识库的个人权限)。
该规则的含义是：用户通过某个组获得的权限，取决于“组在知识库的权限”和“用户在组内的权限”两者的较小值；如果用户属于多个组，则取所有来源中的最大权限；同时再与“用户的个人授权”取最大值。该设计既保证了组权限的约束性，又允许通过个人授权进行补充。系统需要在前端提供可解释的权限展示，例如拆解为“来源A + 来源B → 最终权限”的形式，帮助用户理解复杂场景。
在管理员机制与安全约束方面，系统必须保证每个用户组和每个知识库始终至少存在一个具备管理权限的主体。用户组创建者默认拥有管理权限，知识库创建者同样默认拥有管理权限，且在执行权限变更操作时需进行强校验，防止出现“无管理员”的情况。管理权限用户可以授予他人管理权限，实现权限转移，但系统需在关键操作（如移除管理员、降级权限）时进行拦截与提示。同时建议所有权限变更操作记录审计日志，以支持问题追溯与安全审计。
技术方案上，该模块需要重点解决“权限计算性能、可解释性与一致性”问题：
- 前端：提供用户组管理、成员管理、知识库授权、权限来源展示等完整界面；支持权限可视化（如来源拆解、权限标签）；在危险操作时提供明确提示
- 后端：设计用户组表、用户-组关系表（含权限）、知识库授权表；实现统一权限计算函数，并支持缓存或预计算优化性能；在所有资源接口中接入权限校验中间件；实现权限变更的事务控制与一致性保障
- 算法：作为调用方，仅依赖后端返回的最终权限结果，不参与权限计算逻辑
需要注意：
- 权限计算需高性能实现，建议对高频访问结果进行缓存，并在权限变更时主动失效
- 权限来源必须可解释，否则在多组叠加场景下用户难以理解结果
- P1：“无需确认直接加入组”需要配套通知机制，避免用户无感知加入
- 必须严格防止“最后一个管理员被移除”的情况，所有相关操作需后端强校验
- 当前版本不支持子组嵌套，需在设计与文档中明确该限制，同时在数据模型上预留扩展空间

### D — Personal Mode

#### D1. 整体架构

**需求细化**：
- Personal Mode 是独立的部署形态，与 Server Mode（标准多用户模式）共用同一套代码，通过启动参数 `MODE=personal` 区分。
- 存储替换：
  - PostgreSQL → SQLite（单文件，存放于用户数据目录）
  - OpenSearch → 内置 BM25（基于 SQLite FTS5 或 Tantivy）
  - Milvus → milvus-lite（嵌入式，无需独立进程）
  - Redis → 内存缓存（进程内，重启后失效）
- 去掉 Kong Gateway：前端直接请求 Go core 服务，Go core 服务直接调用 Python algo 服务。
- 去掉 RBAC：所有请求视为 admin 权限，auth-service 简化为本地用户管理（无 OAuth 服务端，仅本地账号）。

#### D2. Electron 打包

**需求细化**：
- 使用 Electron 作为桌面容器，内嵌前端静态资源。
- Go 后端和 Python 算法服务预编译为平台原生二进制（macOS arm64/x86_64、Windows x64、Linux x64）。
- Electron 主进程负责启动/停止 Go 和 Python 子进程，监听子进程崩溃并自动重启（最多重试 3 次）。
- 应用启动时显示加载界面，等待后端服务就绪（健康检查通过）后再展示主界面。
- 数据目录默认为 `~/LazyMind/data/`，支持在设置中修改。
- 支持系统托盘（tray），最小化到托盘后台运行。
- 打包产物：macOS `.dmg`、Windows `.exe` 安装包、Linux `.AppImage`。

#### D3. "新建AI助手"逻辑

**需求细化**：
- Personal Mode 下，"用户"在 UI 层面呈现为"AI助手"：
  - "新建用户" → "新建AI助手"
  - 用户列表页标题 → "我的AI助手"
  - 用户头像区域显示助手名称
- 新建的 AI 助手（即用户）自动加入默认用户组，权限为 `write`。
- 技能页面顶部增加助手切换器（下拉选择），切换后展示对应助手的技能列表。
- 问答页面顶部增加助手切换器，切换后以该助手的身份（技能、记忆、偏好）进行问答。

- Server Mode 下，以上 UI 文案保持原样，不受影响。

#### D4. 每用户事件循环与消息接入（P1）

**需求细化**：
- 为每个用户（助手）维护一个独立的事件队列（基于内存或 SQLite），用于接收外部消息（微信、飞书等）并触发对应的 agent 处理。

- 事件循环在 Personal Mode 下默认启用，Server Mode 下作为可选功能。

### E — Key 管理与模型配置

#### E1. API Key 加密存储与前端脱敏

**需求细化**：
- 所有 API Key 在写入数据库前，使用 AES-256-GCM 加密，密钥由服务端环境变量管理（`KEY_ENCRYPTION_SECRET`），不存入数据库。
- 前端展示 Key 时，仅显示前 4 位 + `****` + 后 4 位（如 `sk-a****xyz`）；创建后不再回显完整 Key。
- 编辑 Key 时，输入框默认为空（不预填脱敏值），用户若不修改则留空，后端收到空值时保持原 Key 不变。
- 后端接口 `GET /api/v1/keys` 返回的 Key 字段统一为脱敏值，不提供获取明文的接口。
- Key 的使用（调用外部 API）在服务端完成，明文 Key 仅在内存中短暂存在，不写入日志。

#### E2. Embedding / MinerU / PaddleOCR Key 配置

**需求细化**：
- 在模型配置页新增"解析服务"分类，包含：
  - MinerU API Key（用于云端 PDF 解析）
  - PaddleOCR API Key（用于 OCR 服务，若使用云端版本）
  - Embedding 模型 Key（若使用外部 embedding 服务）
- 配置策略：
  - 系统级配置（system-admin 设置）：作为全局默认值。
  - 用户组级配置（group-admin 设置）：覆盖系统级，对组内成员生效。
  - 用户级配置：覆盖用户组级，仅对自己生效。
  - 优先级：用户级 > 用户组级 > 系统级。
- 离线解析（MinerU/PaddleOCR）是否允许用户各自配置 Key，待产品确认后补充（当前标记为 TBD）。

#### E3. 核心工具 Key 配置

**需求细化**：
- 在工具配置页（与 K 工具管理联动）新增系统内置工具的 Key 配置，包括：

### G — 问答算法重构与增强

#### G1. Chat / Agentic-Chat 代码重构

**需求细化**：
- 目标：`algorithm/chat/` 目录代码量减少 70%，核心逻辑清晰，算法逻辑与工程胶水代码分离。
- 可复用的算法能力（检索、rerank、记忆注入等）沉淀到 `algorithm/lazyllm/` 的 RAG 工具模块中，以 LazyLLM 组件形式提供。
- `algorithm/chat/pipelines/` 中的 pipeline 定义应只包含流程编排，不包含具体算法实现。
- 重构后需保证现有接口行为不变（输入输出格式兼容）。

#### G2. 包引用规范化

**需求细化**：
- 所有 `algorithm/` 下的 Python 文件，内部引用统一改为相对包引用，顶层包名为 `lazymind`。
- 示例：`from lazymind.algorithm.chat import NaivePipeline`，`from lazymind.algorithm.parsing import DocumentParser`。
- 禁止使用绝对路径（如 `/app/chat/xxx`）或基于 `sys.path` 的 hack。
- 在 `algorithm/` 目录下添加 `pyproject.toml`，将 `lazymind` 注册为可安装包。

#### G3. 删除非流式输出，由后端转换

**需求细化**：
- algo 侧所有 chat 接口统一改为 SSE 流式输出，删除非流式分支代码。
- backend（Go core）在调用 algo 时，若下游请求方需要非流式响应，由 Go 侧消费完整个 SSE 流后，拼接为完整响应再返回。
- 此改动解决非流式模式下 backend→algo 的超时问题（当前 algo 处理时间可能超过 HTTP 超时阈值）。
- Go 侧新增 `StreamToSync` 工具函数，封装 SSE 消费逻辑。

#### G4. Agentic 工具注册与运行时 Key 传递

**需求细化**：
- 工具注册改为动态注册机制：工具定义（名称、描述、参数 schema）在启动时注册，但 Key 等凭证在运行时通过请求上下文传入，不在注册时硬编码。
- 每次 agentic chat 请求携带 `tool_credentials` 字段（加密传输），algo 侧解密后注入对应工具的执行上下文。
- 工具选取逻辑：根据用户在请求中指定的工具列表（或默认开启的工具列表）动态构建工具集，不再全量加载所有工具。

#### G5. 多用户 Key 配置与无 Key 跳过逻辑

**需求细化**：
- algo 侧新增 `KeyResolver` 模块，根据请求中的用户 ID，从 backend 获取该用户的有效 Key（按 E2 的优先级链解析）。
- 对于依赖外部 Key 的功能（rerank、搜索引擎、embedding 等），若 `KeyResolver` 返回空，则：
  - rerank：跳过，直接返回未重排结果。
  - 搜索引擎工具：工具执行返回"未配置搜索引擎 Key，已跳过网络搜索"。
  - embedding：降级为 BM25 检索（若 F2 中配置了降级策略）。
- 跳过行为需在响应的 `metadata` 字段中记录，便于前端展示提示。

#### G6. 飞书知识动态搜索

**需求细化**：
- 用户在知识源配置中，可将飞书知识库设置为"动态模式"（区别于"入库模式"）。
- 动态模式下：
  - 文档不入向量库，不做离线解析。
  - 每次 chat 时，根据 query 实时调用飞书 Search API 检索相关文档片段。
  - 支持对飞书文档的动态修改（通过飞书 API 写回），需用户已配置飞书 OAuth。
- 动态模式和入库模式可共存（同一知识库可同时有入库文档和动态飞书源）。
- 动态搜索结果在 chat 引用来源中标注"飞书实时"标签。

```
Dataset {

### I — 算法自进化

#### I1. 自进化实验报告

**需求细化**：
- 在至少一个公开 RAG 评测数据集（如 BEIR、CMRC 或内部数据集）上完成实验。
- 报告需包含以下对比维度：
  - 基础算法（当前 LazyMind 默认 pipeline）
  - 友商方案（至少一个，如 Dify、FastGPT 等的默认 RAG 配置）
  - 自进化后的算法
- 评测指标：Recall@K、MRR、NDCG、端到端答案准确率（人工评估或 LLM-as-judge）。
- 报告以 Markdown 格式存放于 `docs/experiments/` 目录，包含实验配置、数据集说明、结果表格和结论分析。

#### I2. 自进化控制体验优化

**需求细化**：
- 前端自进化页面（`frontend/src/modules/selfEvolution/`）优化：
  - 展示当前自进化任务的进度（阶段：代码生成 → 沙箱测试 → 冲突检查 → 注册 → 评测 → 灰度）。
  - 每个阶段显示状态（待执行/进行中/成功/失败）和耗时。
  - 支持在任意阶段暂停/取消任务。
- Checkpoint 机制：
  - 每个阶段完成后，将状态持久化到数据库（`evo` 服务的 `session` 表）。
  - 任务中断后（服务重启、网络断开等），重新进入时从最后一个成功的 checkpoint 继续，不从头开始。
  - Checkpoint 数据包含：阶段名、输入/输出快照、时间戳。

#### I3. 新算法上线与切流机制

**需求细化**：

**算法版本管理**：
- 每个算法版本有唯一 ID、版本号、状态（`draft` / `testing` / `active` / `deprecated`）、创建时间。
- 一个知识库可关联多个算法版本，其中一个为"主算法"（`is_primary: true`）。
- 普通用户不可见算法版本信息；管理员在"开发者模式"下可查看和管理。

**上线流程**：
1. **沙箱测试**：自进化生成的新算法代码在隔离容器（Docker sandbox）中运行单元测试，测试通过才进入下一步。
2. **冲突检查**：检查新算法与现有主算法的接口兼容性（输入输出 schema 一致性），无冲突才继续。
3. **注册**：将新算法注册到主服务，状态置为 `testing`。
4. **增量解析**：对基础知识库和被测知识库，使用新算法重新解析（仅处理新增/变更文档，存量文档按需重解析）。
5. **评测**：使用数据集（H 模块提供）对新算法进行自动评测，生成评测报告。
6. **灰度**：若评测效果有提升，进入灰度阶段——配置 router，对指定知识库按比例（如 10%）切流到新算法。
7. **全量上线**：灰度通过后，将新算法设为主算法，对所有知识库生效。
8. **回滚**：30 天内支持一键回滚到上一个主算法；30 天后旧算法版本自动删除（物理删除代码和解析数据）。

**Router 模块**：
- Router 根据配置决定每次 chat 使用哪个算法版本，支持：
  - 按知识库配置（某知识库固定使用某算法）
  - 按流量比例（A/B 测试，如 90% 主算法 + 10% 新算法）
  - 双回复模式（同时调用两个算法，返回两个答案供对比）
- Router 配置由管理员在开发者模式下修改。

### J — 智积阅累（记忆/技能/习惯）

#### J1. Diff 差异对比 UI 升级（P1）

**需求细化**：
- 在记忆/技能/习惯/偏好的版本对比页面，升级 diff 展示：

  - 新增内容用绿色背景标注，删除内容用红色背景+删除线标注。
- **AI 总结变更点**：在 diff 视图顶部，提供"AI 总结"按钮，点击后调用 LLM 生成一段自然语言描述，说明本次变更的主要内容（如"新增了用户对简洁回复风格的偏好，删除了对长篇幅回复的要求"）。

#### J2. 记忆模块版本管理（P1）

**需求细化**：
- 每次记忆/技能/习惯/偏好内容发生变更时，自动创建一个版本快照（存储完整内容，不做 diff 存储）。

- **回退**：用户可选择任意历史版本，点击"回退到此版本"，当前内容替换为该版本内容，并创建一个新版本记录（类型为"回退"）。
- **回收站**：用户删除的记忆条目进入回收站，保留 30 天，支持恢复或永久删除。
- 版本数量上限：每个文档保留最近 50 个版本，超出后自动删除最旧版本。

#### J3. AI 角色/风格配置入口

**需求细化**：
- 在智积阅累设置页新增"AI 个性化"入口，包含：
  - **角色**：自由文本输入，描述 AI 的角色定位（如"你是一个严谨的技术顾问"）。
  - **回复风格**：预设选项（简洁/详细/幽默/正式）+ 自定义文本。
  - **对你的称谓**：AI 称呼用户的方式（如"您"、用户名、自定义昵称）。

#### J4. 自动更新配置（替换"保护态"）

**需求细化**：
- 在技能、记忆、习惯、偏好的每个条目/文档上，增加"是否自动更新"开关（toggle）。
- 各类型默认值：
  - 技能：默认**关闭**（不自动更新）
  - 记忆、习惯、偏好：默认**开启**（自动更新）
  - 词表：无此开关，始终自动更新；有冲突时弹出提醒让用户处理
- 开启自动更新：系统可直接修改内容，用户无感知（但版本历史中有记录）。
- 关闭自动更新：系统只能生成修改建议，需用户在"待审批"列表中手动批准后才生效（联动 N 全局消息/审批机制）。
- 原有"保护态"相关文案全部替换为"自动更新"，UI 上的 toggle 标签为"自动更新"。

#### J5. 建议接受后的 Task 流转机制

**需求细化**：
- 用户点击"接受建议"后：
  1. 立即跳转到"生成草稿"界面（不等待计算完成）。
  2. 页面顶部显示"正在计算中，请稍候..."提示。
  3. 后端异步执行 task，完成后通过 WebSocket 推送状态更新。
- **Task 状态持久化**：
  - Task 状态存储在后端数据库（`evolution/` 模块的 task 表），字段包括：`id`、`user_id`、`doc_id`、`status`（`pending`/`running`/`done`/`failed`/`abandoned`）、`created_at`、`updated_at`。
  - 页面刷新后，前端重新拉取当前文档的 task 状态，恢复到对应界面。
- **Task 生命周期**：
  - 每个文档同时只能有一个 active task（`pending` 或 `running` 状态）。
  - Task 执行失败时，后端自动重试最多 3 次，重试间隔指数退避（1s/2s/4s）。
  - 用户明确"应用"或"放弃"后，task 状态置为 `done`/`abandoned`，才能创建新 task。
- **历史记录**：过往 task 的处理记录（包括失败记录）可在文档的"日志"标签页查看。

#### J6. 记忆更新策略与跨文件整理

**需求细化**：
- **触发条件优化**：记忆更新不再每次对话后都触发，改为以下条件之一满足时触发：
  - 对话轮次达到阈值（默认 10 轮）
  - 对话中出现明确的偏好表达（由 LLM 判断）
  - 用户手动触发"整理记忆"
- **跨文件整理**：新增"整理所有记忆"功能，对用户的所有记忆文件（记忆、习惯、偏好）进行全局去重和整合，消除跨文件的重复 key-point。
- 跨文件整理为异步任务，完成后通知用户（联动 N 全局消息机制）。

#### J7. 时间权重与记忆冲突处理

**需求细化**：
- 每条记忆/偏好条目记录最后更新时间（`updated_at`）。
- 当新的偏好与已有记忆冲突时，系统根据时间权重决策：
  - 新记录（近 7 天内）权重高于旧记录，自动覆盖。
  - 旧记录（超过 30 天）权重低，新记录直接替换。
  - 中间区间（7~30 天）：生成冲突提醒，由用户决定保留哪个版本。
- 冲突提醒通过 N 全局消息机制推送给用户。

#### J8. 避免重复总结与 Key-Point 唯一归属

**需求细化**：
- 系统在生成新的技能/词表/记忆条目前，先检查该 key-point 是否已存在于其他类型中。
- 检查逻辑：使用语义相似度（embedding 相似度 > 0.9 视为重复）判断是否重复。
- 若检测到重复：
  - 不创建新条目。
  - 在已有条目所在类型中更新内容（若有新信息）。
  - 在操作日志中记录"检测到重复，已合并到 [类型] 中的 [条目名]"。

#### J9. 从数据集学习（依赖 H）

**需求细化**：
- 在智积阅累设置页新增"从数据集学习"入口。
- 用户选择一个数据集（H 模块管理的数据集），系统分析数据集中的 query/answer 对，提取：
  - 技能：反复出现的任务模式（如"用户经常问技术方案对比"）
  - 词表：领域专有名词、缩写
  - 记忆：用户偏好（如"用户偏好简洁的回答"）
- 提取结果以建议形式呈现，用户逐条确认后写入对应模块。
- 学习过程为异步任务，完成后通知用户。

### K — 工具管理

#### K1. MCP / RESTful-API 工具配置（P1）

**需求细化**：
- 在设置页新增"工具管理"模块，支持添加以下类型的工具：
  - **MCP 工具**：填写 MCP Server 地址、认证方式（API Key / OAuth）、工具描述。系统自动从 MCP Server 拉取工具列表（`tools/list` 接口）。
  - **RESTful-API 工具**：填写 API 端点、HTTP 方法、请求参数 schema（JSON Schema 格式）、认证方式、工具描述（用于 LLM 理解工具用途）。
- 工具添加后，系统自动验证连通性（发送测试请求），验证失败时提示错误原因。
- 工具列表展示：名称、类型、状态（正常/异常）、最后使用时间。
- 工具的 Key/认证信息按 E1 的加密规则存储。
- 暂不支持 code 工具（自定义代码执行）和 cli 工具（命令行工具），预留扩展点。

#### K2. 工具开启/关闭

**需求细化**：
- 在智积阅累的工具模块，展示所有已配置的工具（系统内置 + 用户自定义），每个工具有开启/关闭开关。
- 关闭的工具在 agentic chat 中不会被选用，即使 LLM 判断该工具适合当前任务。
- 工具开关状态为用户级配置，不影响其他用户。

#### K3. Skill 中引用工具

**需求细化**：
- Skill 文档中支持通过特定语法引用工具（如 `@tool:tool_name`）。
- 执行 skill 时，系统解析引用的工具列表，并在 agentic chat 中自动启用这些工具。
- 若引用的工具已被删除：
  - 执行时忽略该工具引用，不报错中断。
  - 在 skill 编辑页顶部显示警告："此 skill 引用了已删除的工具：[工具名]，请更新 skill 内容。"
  - 不自动删除 skill 中的引用文本，由用户手动清理。

### L — 通用 Skill 与前端插件

#### L0. 插件框架设计

**需求细化**：
- 插件是一种特殊的 UI 组件，在对话中被激活后，其内容（状态）贯穿整个对话上下文。
- 插件激活方式：用户在对话中触发特定 skill，或手动点击工具栏中的插件图标。
- 插件在对话界面以**侧边栏**或**分屏**形式展示，不遮挡对话消息流。
- 插件状态（如 Plan 的任务列表、写作文档内容）持久化到后端，刷新页面后恢复。
- 一个对话中可同时激活多个插件，通过标签页切换。

#### L1. Plan 插件（P1）

**需求细化**：
- 用户可在对话中说"帮我制定一个计划"或手动激活 Plan 插件。
- Plan 插件展示一个任务列表，每个任务有：标题、描述、状态（待完成/进行中/已完成）、负责人（AI 或用户）。
- 用户可手动添加、编辑、删除任务，也可让 AI 根据对话内容自动生成任务列表。
- 每完成一个任务，用户或 AI 可勾选打钩，状态更新为"已完成"。
- AI 在执行任务时，会在对话中引用对应的 Plan 任务（显示任务编号），完成后自动勾选。
- Plan 内容随对话上下文传递，AI 在后续回复中始终感知当前 Plan 状态。

#### L2. AI 写作插件与技能（P1）

**需求细化**：
- **插件 UI**：
  - 分屏展示：左侧对话，右侧文档编辑器。
  - 文档编辑器支持 Markdown 和 LaTeX 两种格式，通过顶部切换。
  - 素材面板（可折叠）：展示大纲、参考图片、规范约束等素材文件。
  - 支持导出为 Word（`.docx`）和 PDF 格式。
- **预置文本类型**（P1）：
  - 内置类型：论文、技术方案、工作报告、标书、小说。
  - 每种类型对应一个 skill 文档，定义撰写流程、素材要求和格式规范。
  - 用户选择类型后，系统自动加载对应 skill，并引导用户提供必要素材。
- **AI 写作流程**：
  1. 用户描述写作需求，AI 生成大纲（素材面板展示）。
  2. 用户确认/修改大纲后，AI 逐节生成正文。
  3. 用户可针对任意段落在对话中提出修改要求，AI 局部修改。
  4. 完成后导出。

#### L3. AI PPT 插件与技能（P1）

**需求细化**：
- **插件 UI**：
  - 分屏展示：左侧对话，右侧 PPT 预览器。
  - PPT 预览器展示幻灯片缩略图列表（左侧）+ 当前页详情（右侧）。
  - 支持拖拽调整页面顺序。
  - 每页展示三个区域：PPT 正文（图片/渲染效果）、设计方案（文字描述）、讲稿。
- **交互修改**：
  - 针对单页：用户可在预览图上圈点标注，说明修改意图，AI 根据标注修改该页。
  - 针对多页：用户指定页码范围，AI 批量修改（如统一风格、字体）。
  - 设计方案和讲稿支持直接在编辑器中修改，也可通过对话描述修改。
- **PPT 生成流程**：
  1. 用户描述 PPT 主题和要求，AI 生成大纲（页数、每页标题）。
  2. 用户确认大纲后，AI 逐页生成内容（正文 + 设计方案 + 讲稿）。
  3. 用户交互修改，完成后导出（格式 TBD：`.pptx` 或 PDF）。
- PPT 渲染方案：后端生成结构化数据（JSON），前端使用 reveal.js 或类似库渲染预览。

### M — 任务看板与可视化（P1）

#### M1. 任务与定时任务

**需求细化**：
- **任务模型**：
  ```
  Task {
    id: string
    title: string
    description: string
    type: enum           // 'manual' | 'scheduled' | 'agent_triggered'
    status: enum         // 'pending' | 'running' | 'done' | 'failed' | 'cancelled'
    owner_id: string     // 创建者/负责人
    agent_id: string     // 处理该任务的 agent（可为空，表示人工任务）
    cron_expr: string    // 定时任务的 cron 表达式（可为空）
    next_run_at: timestamp
    created_at: timestamp
    updated_at: timestamp
  }
  ```
- **定时任务**：支持通过 cron 表达式配置定时触发，触发后自动创建一个 `agent_triggered` 类型的任务实例，分配给对应 agent 处理。
- 定时任务管理页：展示所有定时任务，支持启用/禁用、立即触发、查看历史执行记录。

#### M2. 三省六部 Agent 任务看板

**需求细化**：
- 任务看板以**看板视图**（Kanban）展示，列为任务状态（待处理/进行中/已完成/失败）。
- 每个任务卡片展示：标题、负责 agent、优先级、创建时间、当前状态。
- "三省六部"是 agent 的组织架构隐喻，具体 agent 角色定义（如"礼部"负责对外沟通、"工部"负责文档处理等）由产品进一步定义，此处预留 `agent_role` 字段。
- 看板支持按 agent 角色筛选，支持拖拽任务卡片在列之间移动（手动更新状态）。
- 看板数据实时更新（WebSocket 推送）。

#### M3. 任务处理情况查看

**需求细化**：
- 点击任务卡片，展开任务详情侧边栏，包含：
  - 任务基本信息（标题、描述、状态、时间线）
  - **Agent 处理记录**：按时间顺序展示 agent 的每一步操作（思考过程、工具调用、中间结果），类似 ReAct 链的可视化。
  - 若任务失败，展示失败原因和错误堆栈（脱敏后）。
- 处理记录支持折叠/展开，默认折叠中间步骤，只展示关键节点。

### N — 全局消息处理/审批机制（P1）

#### N1. 消息中心

**需求细化**：
- 在界面右上角新增"消息中心"入口（铃铛图标），显示未读消息数量角标。
- 消息类型：
  - **系统通知**：服务状态变更、任务完成/失败等。
  - **待审批**：需要用户手动批准的操作（如关闭自动更新后的记忆修改建议、J4 中的建议审批）。
  - **冲突提醒**：记忆冲突（J7）、词表冲突等需要用户决策的情况。
  - **Agent 消息**：agent 主动发起的通知或请求（如"我发现了一个可能有用的信息，是否加入记忆？"）。
- 消息列表支持按类型筛选，支持标记已读/全部已读。

#### N2. 审批流程

**需求细化**：
- 待审批消息包含：操作描述、变更内容预览（diff 形式）、操作来源（哪个 agent/系统模块发起）。
- 用户操作：**批准**（变更立即生效）、**拒绝**（变更丢弃，记录拒绝原因可选）、**修改后批准**（用户可编辑变更内容后再批准）。
- 审批操作记录到操作日志，包含操作人、时间、决策结果。
- 批量审批：支持勾选多条待审批消息，批量批准或拒绝。
- 超时处理：待审批消息超过 7 天未处理，自动标记为"已过期"，不再生效（不自动批准）。

### O — 浏览器插件（P2）

#### O1. 插件整体设计

**需求细化**：
- 浏览器插件支持 Chrome/Edge（Manifest V3）。
- 插件激活方式：点击浏览器工具栏图标，或使用快捷键（默认 `Ctrl+Shift+L`）。
- 激活后，在当前页面右侧弹出对话侧边栏（宽度约 400px），不影响原页面布局。
- 插件需配置 LazyMind 服务地址（Personal Mode 下默认为 `localhost`），首次使用时引导配置。

#### O2. 页面信息注入

**需求细化**：
- 插件自动提取当前页面的关键信息，注入对话上下文：
  - 页面标题、URL
  - 页面主体文本内容（去除导航、广告等噪声，使用 Readability.js 提取）
  - 若页面有结构化数据（如飞书文档 ID、Notion 页面 ID），额外提取。
- 提取的页面信息以"当前页面上下文"形式附加到每次对话请求，用户可在侧边栏中查看和编辑注入的内容。

#### O3. 飞书页面读写支持

**需求细化**：
- 若当前页面为飞书文档/多维表格/知识库，插件识别页面类型，并在侧边栏顶部显示"飞书文档已识别"标签。
- 若用户已配置飞书 OAuth（在 LazyMind 设置中），插件可：
  - **读取**：获取文档完整内容（通过飞书 API）。
  - **写入**：根据 AI 生成的内容，调用飞书 API 修改文档（需用户在侧边栏中确认操作）。
- 若用户未配置 OAuth：
  - 插件仅能读取页面可见文本（不调用飞书 API）。
  - 当用户发出写入指令时，提示："需要先配置飞书授权，点击此处前往配置。"

#### O4. 不支持能力的处理

**需求细化**：
- 对于非飞书等不支持写入的页面（如淘宝、微博等）：
  - 插件仅读取页面内容，不提供写入能力。
  - 若用户发出修改/操作指令，AI 回复："我目前只能读取此页面的内容，无法对该网站进行操作。"
  - 读取的内容可用于 AI 分析（如商品评价分析、联网搜索补充信息）。
- 插件不主动声明"支持/不支持"某网站，而是在用户尝试操作时给出明确反馈。
